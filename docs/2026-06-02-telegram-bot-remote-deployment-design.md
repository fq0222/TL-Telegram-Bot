# Telegram 机器人独立服务器部署架构设计

## 背景

在已有 Telegram 机器人接入方案基础上，进一步规划“机器人部署在独立服务器”的架构方案。该方案的核心目标，是让 Telegram 机器人从当前业务后端进程中解耦出来，形成独立运行、独立发布、独立运维的接入层，同时保持业务规则仍然以主业务服务为唯一真相来源。

当前管理端仅在本地或局域网访问，没有暴露到公网。本设计需要明确：

1. 独立部署机器人时，是否需要开放管理端页面。
2. 主业务服务与机器人服务之间如何通信。
3. 每个阶段的能力应由哪个系统负责。

## 结论

独立部署机器人时，不需要把管理端页面暴露到公网。

机器人真正需要访问的是主业务服务提供的“机器人专用内部 API”，而不是 `client-admin` 页面。推荐做法是：

1. 管理端仍然保持本地或局域网访问。
2. 主业务服务额外提供内部 API。
3. 机器人服务器通过受控网络访问内部 API。
4. 机器人再对接 Telegram Bot API。

## 总体架构

独立部署后，系统建议拆分为两个服务：

### 主业务服务

部署在当前业务服务器，负责：

- 用户、订单、套餐、订阅、流量等业务真相
- 3X-UI 集成与健康检查
- 数据库存取
- 绑定码生成与绑定关系管理
- 告警状态、命令审计、敏感操作审计
- 机器人专用内部 API

### Telegram Bot 服务

部署在独立服务器，负责：

- 对接 Telegram Bot API
- 轮询或 webhook 接收消息
- 解析管理员和用户命令
- 调用主业务服务内部 API
- 将结果格式化后返回给 Telegram
- 转发告警消息

## 为什么必须走 API，而不是机器人直接连数据库

不建议机器人直接连接主库，也不建议在机器人项目里复制业务逻辑。

推荐架构是：

- 主业务服务保留全部业务规则
- 机器人只做外部接入层和消息分发层

这样做的好处：

1. 网站与机器人共用一套业务逻辑，不会发生逻辑漂移。
2. 业务规则仍然在现有项目内维护，避免双份实现。
3. 机器人服务故障不会直接污染主站数据库访问层。
4. 后续若扩展飞书、钉钉、企业微信，也可继续复用同一套内部 API。

## 网络拓扑建议

推荐以下网络模型之一：

### 方案一：VPN 或零信任内网互通

主业务服务器与机器人服务器通过：

- Tailscale
- ZeroTier
- WireGuard

实现专用内网互通。

优点：

- 安全性高
- 不需要把内部 API 完全暴露到公网

### 方案二：公网白名单访问

主业务服务的内部 API 暴露到公网，但仅允许机器人服务器 IP 访问。

优点：

- 部署简单

缺点：

- 依赖固定出口 IP
- 暴露面比内网方案大

### 方案三：反向隧道或零信任代理

通过：

- Cloudflare Tunnel
- Nginx 反代 + 鉴权

把内部 API 受控暴露给机器人访问。

## 管理端是否需要暴露公网

不需要。

因为机器人独立部署时访问的是：

- 主业务服务提供的内部 API

而不是：

- 管理端页面

也就是说：

1. `client-admin` 不需要为机器人开放公网访问。
2. 真正需要对外可达的是一组受控的后端内部接口。
3. 管理后台是否公网可访问，和 Telegram 接入没有直接必然关系。

## 分阶段架构设计

## 第一阶段：管理员巡检与告警

### 目标

让独立机器人先成为运维告警和管理员状态查询入口。

### 业务边界

#### 主业务服务负责

- 执行或汇总 3X-UI 巡检
- 存储服务器健康状态
- 存储故障、恢复、发送状态
- 生成管理员绑定码
- 校验管理员绑定码
- 提供服务器状态与告警查询接口
- 提供管理员代查用户流量概览接口

#### 机器人服务负责

- 对接 Telegram Bot API
- 管理 `/bind` `/status` `/servers` `/server` `/alerts` `/user`
- 调用主业务服务内部 API
- 将待发送告警推送给管理员

### 阶段一推荐通信方式

推荐“机器人主动拉取待发送告警”模式。

即：

1. 主业务服务只负责写告警记录。
2. 机器人服务定时拉取待发送告警。
3. 机器人发送完成后回执主业务服务。

优点：

- 主业务服务不必主动连机器人服务器
- 网络方向单一，更适合独立部署
- 主业务服务更干净

## 第二阶段：用户自助查询

### 目标

在阶段一稳定后，开放用户绑定和自助查询。

### 业务边界

#### 主业务服务负责

- 生成用户绑定码
- 校验绑定码
- 建立用户与 Telegram chat 的绑定关系
- 提供用户流量、套餐、订阅、节点摘要查询接口
- 保留管理员代查接口

#### 机器人服务负责

- 管理 `/me` `/usage` `/subscription` `/nodes` `/unbind`
- 识别当前 chat 绑定的用户
- 调用主业务服务获取用户信息
- 返回格式化文本

### 阶段二核心原则

阶段二仍以只读为主，不在机器人侧保存业务真相。机器人可以保存少量会话状态，但不能成为业务主数据源。

## 第三阶段：受控写操作

### 目标

开放部分高价值但高风险的受控操作。

### 业务边界

#### 主业务服务负责

- 所有真正的写操作
- 风险校验
- 二次确认码生成与校验
- 审计日志
- 频率限制
- 敏感命令权限控制

#### 机器人服务负责

- 命令入口
- 展示确认步骤
- 收集确认码
- 调用主业务服务完成写操作
- 返回结果给用户或管理员

### 阶段三核心原则

机器人不能直接执行写操作逻辑，只能发起请求；真正的数据修改必须由主业务服务完成。

## 服务间鉴权设计

独立部署下，主业务服务与机器人服务之间必须增加服务级鉴权。

推荐请求头：

- `X-Internal-Client: telegram-bot`
- `X-Internal-Timestamp`
- `X-Internal-Signature`

签名建议使用：

- `HMAC-SHA256(secret, method + path + timestamp + body)`

这样可以避免单个固定 token 被复制后长期滥用。

## 业务级鉴权设计

即使服务间鉴权通过，主业务服务仍需在每个业务接口中继续判断：

1. 当前 `chat_id` 是否已绑定。
2. 当前绑定身份是管理员还是普通用户。
3. 当前身份是否有权访问目标资源。

不能只依赖机器人侧判断。

## 可靠性设计

### 告警可靠性

阶段一建议主业务服务记录：

- 待发送状态
- 最近发送时间
- 发送次数
- 是否已恢复

机器人拉取后发送成功，再回执主业务服务，避免重复轰炸。

### 机器人故障隔离

机器人服务异常时：

- 不影响主业务接口
- 不影响主业务数据库
- 告警记录仍保存在主业务服务中，待机器人恢复后继续发送

### API 失败重试

机器人调用主业务服务内部 API 时应做：

- 网络超时重试
- 429/5xx 有限重试
- 明确失败日志

## 数据流设计

### 管理员绑定流

1. 管理员在本地或局域网访问的管理端点击生成绑定码。
2. 管理端调用主业务服务接口生成绑定码。
3. 管理员在 Telegram 中发送 `/bind <code>`。
4. 机器人服务调用主业务服务校验绑定码。
5. 主业务服务保存绑定关系。
6. 机器人回复绑定成功。

### 告警发送流

1. 主业务服务巡检后发现故障。
2. 主业务服务生成或更新告警记录为待发送。
3. 机器人轮询拉取待发送告警。
4. 机器人向管理员发送 Telegram 消息。
5. 机器人回执发送结果给主业务服务。
6. 主业务服务更新告警记录发送状态。

### 用户查询流

1. 用户发送 `/usage`。
2. 机器人识别 chat_id。
3. 机器人调用主业务服务用户查询接口。
4. 主业务服务返回当前用户可见数据。
5. 机器人格式化并返回消息。

### 高风险写操作流

1. 用户发送 `/change_password`。
2. 机器人调用主业务服务发起改密请求。
3. 主业务服务创建一次性确认流程。
4. 用户提交确认码和新密码。
5. 机器人将请求转发到主业务服务。
6. 主业务服务完成校验、写审计、修改密码并返回结果。

## 阶段化实施建议

## 第一阶段实施建议

优先完成：

1. 机器人内部 API 鉴权
2. 管理员绑定
3. 服务器健康状态查询
4. 待发送告警拉取
5. 告警发送回执
6. 管理员代查用户流量概览

## 第二阶段实施建议

在阶段一稳定后追加：

1. 用户绑定
2. 用户信息查询
3. 流量查询
4. 订阅信息查询
5. 节点摘要查询

## 第三阶段实施建议

最后追加：

1. 改密码
2. 重新生成订阅
3. 触发重检
4. 提醒配置
5. 更严格的审计和限流

## 安全建议

1. 内部 API 不要直接裸露给所有公网访问。
2. 尽量采用 VPN、零信任网络或 IP 白名单。
3. 内部 API 和普通前端 API 分路径隔离。
4. 高风险写操作必须有二次确认。
5. 日志中不要输出 Bot Token、签名秘钥、确认码。

## 总结

独立服务器部署机器人的最佳实践是：

1. 机器人不直接连接数据库。
2. 机器人不复制业务逻辑。
3. 机器人通过主业务服务提供的内部 API 访问业务能力。
4. 管理端页面不需要暴露公网。
5. 第一阶段先做管理员告警闭环，第二阶段再加用户查询，第三阶段最后做高风险写操作。

这样既能保留当前项目的业务边界，又能为未来多渠道机器人接入预留清晰扩展点。
